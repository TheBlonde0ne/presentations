The Toolbox

MCP, Skills & Subagents

Model Context Protocol (MCP)

A universal standard for connecting agents to tools

  • Created by Anthropic (Nov 2024)
  • Adopted by OpenAI, Google, Microsoft
  • Donated to Linux Foundation (Dec 2025)
  • Growing open-source ecosystem

Example MCP servers:

  • Context7 — fetches up-to-date library docs so the agent never guesses APIs
  • GitHub — create issues, open PRs, review code, manage repositories
  • Database — query SQL/NoSQL databases, inspect schemas, run migrations
  • Filesystem, Zotero, Slack, …
MCP is the real game changer. Before MCP, every integration was bespoke. Now there's one standard. Think of it like LSP for editors, but for AI tools. Every major AI company has adopted it.

Anatomy of an MCP Server

from fastmcp import FastMCP

mcp = FastMCP("Demo")

@mcp.tool
def add(a: int, b: int) -> int:
    """Add two numbers"""
    return a + b

FastMCP — framework handles protocol, transport, and discovery

FastMCP("Demo") — server name for agent discovery

@mcp.tool — decorator exposes the function to any connected agent

Type annotations — agent knows what inputs to provide (self-descriptive)

Docstring — agent reads this to decide when to call the tool

7 lines of Python to give any agent a new capability

Anatomy of a Skill

# skills/brainstorming/SKILL.md

---
name: brainstorming
description: Explore intent and
  requirements before coding
---

# Brainstorming

Before writing any code:
1. Ask what the user wants to achieve
2. Explore edge cases and constraints
3. Propose 2–3 approaches
4. Get explicit approval before proceeding

Just a Markdown file — no code, no server, no build step

name — identifies the skill for discovery and invocation

description — tells the agent what this skill does and when to use it

Instructions — natural language steps the agent follows

Similar to skills, there are also subagent prompts — reusable instructions for spawning specialized subagents (not covered here).

MCP vs. Skills + Command Line Interface

Two ways to extend what an agent can do

MCP Server

A running service the agent calls via tool protocol

  • Self-descriptive tools
  • Persistent process
  • Can be served remotely
  • More overhead to set up

Skill + Command Line Interface

Markdown instructions + existing command line tools

  • Skill describes how to use a command line tool
  • No running process needed
  • Easy to test (works for humans too)
  • Full flexibility of shell commands

Both achieve similar results — the agent gets new capabilities either way

The Context Window Problem

Agents have a limited working memory — long sessions fill it up

Each compaction loses information — early decisions, reasoning, constraints

Mitigations

Subagents — do focused work in their own context, report back a short summary

Spec files — persist decisions, constraints, and plans to disk so they survive compaction

Memory — agent reads/writes key facts to a persistent store it can consult later

Prompting Agents: Dos & Don’ts

Vague

  • “Fix my code”
  • “Refactor the project”
  • “Write tests”
  • “Make it better”

Forces the agent to guess scope, location, and approach — leads to hallucination

Precise

  • “The test in test_forces.py:42 fails — the force loop doesn’t apply periodic boundaries”
  • “Refactor simulation/integrator.py: extract the thermostat into its own class”
  • “Add tests for the neighbor list using TDD — write tests first, then implement”

Rule of thumb: reduce what the agent has to assume

Which file? Which function? What’s wrong? How should it be fixed?
The more precise you are, the better the result.
